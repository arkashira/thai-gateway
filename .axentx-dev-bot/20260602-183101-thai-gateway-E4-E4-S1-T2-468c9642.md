**Title:** Thai‑Gateway Real‑Time API Anomaly Detection & Remediation (T‑GARD)

---

### 1. Problem Statement  
Thailand’s digital economy is rapidly expanding, with the Thai‑Gateway platform serving as the central API hub for government services, e‑commerce, and fintech. While the gateway enforces basic rate limits, it lacks a dynamic, behavior‑driven anomaly detection layer that can identify malicious or cost‑draining usage patterns (e.g., credential stuffing, sudden traffic spikes, atypical endpoint sequences) in real time. Existing monitoring tools focus on static thresholds or post‑mortem log analysis, leaving a critical gap for early detection and automated remediation of abusive traffic that can compromise revenue, security, and service availability.

---

### 2. Target Audience  
- **Thai‑Gateway Platform Engineers** – responsible for maintaining API stability and security.  
- **Government Service Providers** – need to protect citizen data and ensure uninterrupted service.  
- **Fintech & E‑commerce Operators** – rely on the gateway for transaction APIs and require rapid abuse detection.  
- **Security Operations Centers (SOC)** – require actionable alerts for credential abuse and fraud.  
- **API Management Teams** – need an embeddable module that integrates with the existing gateway architecture.

---

### 3. Solution Overview  
T‑GARD is a lightweight, gateway‑native anomaly detection engine that ingests live API telemetry from the Thai‑Gateway, applies a hybrid detection model, and triggers automated remediation actions. It is designed to be deployed as a side‑car or micro‑service within the existing gateway stack, ensuring zero‑touch integration and minimal latency impact.

Key components:

1. **Telemetry Collector** – captures request metadata (endpoint, client ID, auth token, IP, geo, latency, payload hash) from the gateway’s request pipeline and forwards it to a Kafka topic.  
2. **Feature Store** – Redis Streams + ClickHouse aggregate per‑client, per‑endpoint metrics with millisecond granularity.  
3. **Detection Engine** – FastAPI micro‑service running:
   - **Statistical Baselines** (moving averages, percentiles) per endpoint/client.  
   - **Unsupervised ML** (Isolation Forest, Auto‑Encoder) for multi‑dimensional outlier detection.  
   - **Rule‑Based Engine** for known Thai‑Gateway abuse patterns (e.g., rapid token reuse, geo‑hopping across provinces).  
4. **Action Engine** – Serverless functions (AWS Lambda / GCP Cloud Functions) that:
   - Trigger gateway‑level throttling or token revocation via the gateway’s admin API.  
   - Send real‑time alerts to Slack, PagerDuty, or the gateway’s SOC dashboard.  
5. **Dashboard & Feedback Loop** – React + D3 UI hosted on the gateway’s internal portal, allowing analysts to label anomalies and feed labels back into nightly model retraining.

## 1. Acceptance Criteria  
1. **Endpoint Availability** – `GET /metrics` returns HTTP 200 with JSON body.  
2. **Global Request Counters** – JSON contains `total_requests`, `successful_requests`, `failed_requests`.  
3. **Latency Percentiles** – JSON contains numeric `latency_p90` and `latency_p99` (milliseconds).  
4. **Provider‑Specific Rate‑Limit Usage** – JSON contains a `providers` object where each key is a provider name and each value includes `current_usage` and `limit`.  
5. **No Sensitive Data** – The JSON payload never contains request prompts, user IDs, or any field that could leak private information.  
6. **Real‑Time Updates** – Metrics reflect the most recent request within < 1 second after the request completes.  
7. **Thread‑Safety** – Concurrent requests from multiple workers do not corrupt counters or percentile calculations.  
8. **Rate‑Limit on Metrics Endpoint** – The `/metrics` endpoint itself is protected by a global rate‑limit (e.g., 5 req/s) to prevent abuse.

import pytest
from src.providers.rate_limiter import RateLimiter

@pytest.fixture
def limiter():
    return RateLimiter(provider="test-provider", limit=100)

def test_record_and_metrics(limiter):
    # Record a mix of successes/failures with varying latencies
    limiter.record_request(success=True, latency_ms=50)
    limiter.record_request(success=False, latency_ms=120)
    limiter.record_request(success=True, latency_ms=80)

    metrics = limiter.get_metrics()

    assert metrics["total_requests"] == 3
    assert metrics["successful_requests"] == 2
    assert metrics["failed_requests"] == 1
    # Percentiles are calculated over [50, 120, 80]
    assert metrics["latency_p90"] == 120
    assert metrics["latency_p99"] == 120
    assert metrics["current_usage"] == 3
    assert metrics["limit"] == 100

def test_thread_safety(limiter):
    import threading
    def worker():
        for _ in range(1000):
            limiter.record_request(success=True, latency_ms=10)

    threads = [threading.Thread(target=worker) for _ in range(10)]
    for t in threads: t.start()
    for t in threads: t.join()

    metrics = limiter.get_metrics()
    assert metrics["total_requests"] == 10_000
    assert metrics["successful_requests"] == 10_000
