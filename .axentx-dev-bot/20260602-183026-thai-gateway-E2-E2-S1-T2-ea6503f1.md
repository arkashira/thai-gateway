# Diagnosis
The current gateway implementation does not check for the `X-Translate` header, resulting in translation being performed unconditionally. The solution requires adding a header check to enable/disable translation based on the presence and value of the header.

# Proposed change
Modify `/opt/axentx/thai-gateway/src/handler.rs` to:
1. Extract the `X-Translate` header from the request.
2. Set a flag to skip translation if the header is present and equals `"false"`.
3. Conditionally call the translation model based on this flag.

@@ -9,6 +12,14 @@ pub async fn handle(request: Request) -> Response {
     // Check for translation toggle header
     let skip_translation = request.headers()
         .get("X-Translate")
+        .map(|v| v.as_str() == "false")
+        .unwrap_or(false);

 }

## 1. Acceptance Criteria
1. **Default translation** – If the incoming HTTP request **does not contain** the `X-Translate` header, the gateway must invoke `translate_prompt` and the response body must contain the Thai‑translated prompt.  
2. **Explicit disable** – If the request includes `X-Translate: false` (case‑insensitive), the gateway must **skip** `translate_prompt` and forward the original English prompt unchanged to the AI provider; the response body must contain the original prompt.  
3. **Explicit enable / any other value** – If the request includes `X-Translate: true` **or any value other than** the literal `"false"` (e.g., `"yes"`, `"1"`), the gateway must perform translation exactly as in the default case.  
4. **Error propagation** – Any error returned by `translate_prompt` or `ai_provider_call` must be propagated unchanged (via `?`) resulting in an HTTP 5xx response with the original error payload.  
5. **Performance bound** – When translation is performed, the total latency added by `translate_prompt` must be **≤ 0.5 s** (measured in a controlled test harness).  
6. **Compile‑time safety** – The code must compile with `cargo test --all` and `cargo clippy` without warnings or panics.
