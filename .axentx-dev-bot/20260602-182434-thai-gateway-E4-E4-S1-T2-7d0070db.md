2. **Hot‑reload** – When `gateway.yaml` is edited on disk, the gateway detects the change and reloads the configuration within **5 seconds**.  
3. **Atomic update** – The new configuration replaces the old one only after successful validation; in‑flight requests continue to use the previous config until the switch.  
4. **Validation errors** – Invalid YAML (syntax error, missing required fields, unknown provider, wrong data types) aborts the reload and returns a **clear, user‑friendly error** logged to stdout/stderr and exposed via the `/health/config` endpoint (HTTP 200 with JSON `{status:"error",msg:"…"}` ).  
5. **Graceful fallback** – If validation fails, the gateway continues running with the last known good configuration.  
6. **Security** – No external files are read/written beyond the configured `gateway.yaml`; file‑watcher runs with least‑privilege permissions.  
7. **Observability** – Each successful reload emits a structured log entry (`event:"config_reload", version:<hash>, duration_ms:<int>`).
