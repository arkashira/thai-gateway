The goal is to create a robust, real-time load balancer that intelligently routes requests to the least-congested provider while handling failures gracefully.

#### 1. Diagnosis & Core Issues

The current implementation suffers from several critical flaws:
*   **Static Routing:** It uses a simple round-robin without considering the real-time performance and error rates of each provider.
*   **No Dynamic Metrics:** Provider load is not tracked dynamically; it's either static or a simple counter, which doesn't reflect actual performance.
*   **Lack of Configurability:** The retry logic and routing strategy are hard-coded and not exposed via environment variables.
*   **No Automatic Retries:** The system assumes every request succeeds and does not attempt to retry on failure.

#### 2. Proposed Solution

We will implement a dynamic load balancer that:
*   **Uses Real-Time Metrics:** Selects the provider with the lowest combined load score, calculated from response time and error rate.
*   **Supports Configurable Retries:** Allows the number of retries and other parameters to be set via environment variables.
*   **Handles Failures:** Automatically retries on failure, switching to the next best provider if the current one fails.

### 1. **Acceptance Criteria**
- **Congestion Detection** – The proxy must expose a metrics endpoint that returns current response time and error rate for each provider, and use these metrics to rank providers.
- **Round‑Robin with Least‑Congested Selection** – Incoming AI requests are routed to the provider with the lowest combined congestion score (e.g., weighted sum of response time and error rate). If multiple providers tie, round‑robin order is preserved.
- **Automatic Retry** – If a request to the selected provider fails (non‑2xx status or network error), the proxy automatically retries the request on the next‑best provider in the ranking list, up to a configurable maximum retry count.
- **Environment‑Driven Configuration** – The following environment variables must be respected:
  - `PROVIDER_URLS` – comma‑separated list of provider base URLs.
  - `MAX_RETRIES` – maximum number of retry attempts per request.
  - `CONGESTION_WEIGHT_RT` & `CONGESTION_WEIGHT_ER` – weights for response time and error rate in the congestion score.
- **Logging** – Each routing decision and retry attempt must be logged with provider URL, request path, status code, and timestamp.
