## 1. Acceptance Criteria
1. **Retry Count** – When a request to a provider fails, the gateway retries the same provider **exactly 3 times** before giving up.  
2. **Fallback Routing** – After the 3 retries fail, the request is automatically forwarded to the **next healthy provider** in the configured list.  
3. **Error Logging** – Every failed request (each retry attempt and the final failure) is logged via `logger.Error` with:
   - ISO‑8601 timestamp  
   - Provider identifier  
   - HTTP method & URL  
   - Error message / status code  
4. **Exponential Backoff** – Retry intervals follow a backoff pattern (e.g., 100 ms, 200 ms, 400 ms) and are **bounded** (max ≤ 2 s).  
5. **No Duplicate Requests** – The gateway must **not** send more than one request to the fallback provider for the same original client request.  
6. **Observability** – A metric `gateway_provider_retries_total` increments per retry; a metric `gateway_fallbacks_total` increments per fallback.  
7. **Performance** – Under normal operation (no failures) the added logging/retry code adds **≤ 5 ms** latency per request (measured on a warm container).
