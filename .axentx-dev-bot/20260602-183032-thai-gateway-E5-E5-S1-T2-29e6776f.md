## Test Plan – Thai Gateway : Platform‑Specific Rate Limiting

### 1. Acceptance Criteria
1. **API‑Key Generation** – On the first registration request (`POST /register`) the system returns a newly generated API key that is persisted and uniquely associated with the requesting platform.  
   *Metric*: `apiKey` matches pattern `^pk_[a-f0-9]{32}$` and is stored in the `platforms` table.
2. **Key Scope Isolation** – An API key can only access resources belonging to its own platform. Any request that supplies a key belonging to a different platform must be rejected with **HTTP 401 Unauthorized**.  
   *Metric*: 401 response for cross‑platform access attempts.
3. **Per‑Platform Rate Limit** – Each platform is limited to **120 requests per minute** (default, configurable per platform). Exceeding the limit returns **HTTP 429 Too Many Requests** with a `Retry-After` header indicating seconds until reset.  
   *Metric*: 429 after 121st request within a 60 s window; `Retry-After` ≈ remaining seconds.
4. **Independent Quotas** – Rate limits are enforced **per platform**; traffic from Platform A must not affect the quota of Platform B.  
   *Metric*: Platform B can still make 120 requests while Platform A is throttled.
5. **Rate‑Limit Reset** – After the 60‑second window expires, the quota resets and the platform can again make the full number of requests.  
   *Metric*: After advancing time > 60 s, a previously throttled platform receives HTTP 200 for a new request.
6. **Documentation** – `README.md` contains an “Onboarding & Rate Limiting” section that describes:
   - Registration endpoint and payload
   - Retrieval and storage of the API key
   - Default rate‑limit values and how to override them
   - Expected HTTP status codes for success, auth failure, and throttling.  
   *Metric*: Presence of the section (checked via a simple file‑search test).
