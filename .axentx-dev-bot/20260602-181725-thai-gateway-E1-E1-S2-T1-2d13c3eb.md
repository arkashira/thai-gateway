		return

## 1. Acceptance Criteria
1. **200 OK on valid request** – When a request contains a well‑formed JSON body with both string fields `prompt` and `model`, the endpoint returns **200** and a JSON payload.
2. **Response schema** – The 200 payload must contain the keys  
   `id` (RFC‑4122 UUID string), `model` (echoed request value), `provider` (non‑empty string), `usage` (object with numeric `prompt_tokens` and `completion_tokens`), and `response` (string). No extra top‑level fields are required.  
3. **400 Bad Request on validation errors** – Missing, empty, or non‑string `prompt`/`model` fields result in **400** with a JSON error object `{ "error": "<description>" }`.  
4. **429 Too Many Requests on provider exhaustion** – If the internal provider pool reports that *all* providers are exhausted, the endpoint returns **429** with `{ "error": "All providers exhausted" }`.  
5. **415 Unsupported Media Type when Content‑Type ≠ application/json** – The handler must reject non‑JSON bodies before parsing.  
6. **Idempotent request‑id generation** – Each successful call generates a unique UUID; duplicate payloads must not reuse the same `id`.  
7. **Performance bound** – Under a simulated load of 50 concurrent requests, 95 % of successful responses must be returned within **500 ms** (measured in the integration test harness).
