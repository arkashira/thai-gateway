## 1. Acceptance Criteria
1. **YAML location & loading** – On startup the gateway reads a file named `gateway.yaml` located at the project root (or a path supplied via `CONFIG_PATH` env var) and does **not** panic; it returns a `Result<Config, ConfigError>`.
2. **Schema validation** – The file must contain a top‑level key `providers` that is an array of objects. Each object **must** have exactly the fields:
   - `id` – non‑empty string, unique across providers.  
   - `url` – syntactically valid absolute URL.  
   - `api_key` – non‑empty string.  
   - `quota` – integer ≥ 0.  
   Missing fields, extra fields, duplicate `id`s, or invalid values cause a **startup validation error**.
3. **Error logging** – Validation errors are logged at **error** level with a message that includes the provider index (0‑based) and the specific problem (e.g., “provider[1] missing `api_key`”).
4. **Successful load** – When the YAML is valid, `Config::load()` returns a `Config` struct whose `providers: Vec<Provider>` exactly mirrors the YAML content (order preserved).
5. **Sample config** – A file `samples/gateway.yaml` is committed, follows the schema, and is referenced in the README.
6. **Thread‑safety & re‑entrancy** – The loader can be called multiple times concurrently (e.g., from different tests) without sharing mutable global state or leaking file handles.
