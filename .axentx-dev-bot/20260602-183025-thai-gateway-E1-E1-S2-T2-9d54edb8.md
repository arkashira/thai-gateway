## 1. Acceptance Criteria
1. **Health endpoint** – `GET /health` returns HTTP 200 with body exactly `{"status":"ok"}` and `Content‑Type: application/json`.
2. **Chat endpoint success** – `POST /api/v1/chat` with a valid JSON payload `{ "prompt": "<string>" }` returns HTTP 200, `Content‑Type: application/json`, and a JSON body containing a non‑empty `response` field.
3. **Invalid payload** – `POST /api/v1/chat` with missing or non‑string `prompt` returns HTTP 400 with an error JSON `{ "error": "Invalid payload" }`.
4. **Downstream AI timeout** – If the internal AI service does not respond within **5 seconds**, the gateway returns HTTP 504 with JSON `{ "error": "AI service timeout" }`.
5. **Graceful cancellation** – When a timeout occurs, the downstream request is aborted and no resources are leaked (e.g., open sockets, tasks).
6. **Logging** – Each request (health, chat, timeout, bad payload) logs a structured entry with request ID, endpoint, status code, and latency.
7. **Concurrency** – The gateway can handle at least 50 concurrent chat requests without exceeding a 200 ms average latency for successful responses (excluding AI latency).
