## TL;DR  
Add a **self‑contained health‑check subsystem** to the Thai‑Gateway service that  

1. **Registers every AI provider** (name + base URL) once at start‑up.  
2. **Runs a safe, concurrent ticker** that pings each provider’s `/health` endpoint every 30 s (5 s timeout).  
3. **Keeps a thread‑safe map** of `ProviderHealth` (healthy flag, last‑check timestamp).  
4. **Exposes `/health`** (JSON) so ops can see the status of every provider.  
5. **Integrates with the request router** – the router reads the map and **skips any provider that is currently marked unhealthy**, falling back to the next healthy one.  

All of this lives in a single new file `service/health.go` (no external cron library needed) and a few tiny changes to the existing router and `main.go`. The implementation is fully unit‑testable and can be verified with `go test ./...`.

---

## 1. Design Decisions (why we chose these bits)

| Requirement | Candidate 1 | Candidate 2 | Our Choice & Reason |
|-------------|-------------|-------------|---------------------|
| **Scheduler** | Uses `robfig/cron` (extra dependency). | Uses `time.Ticker` (standard lib). | **Ticker** – no extra module, easier to reason about, fits Go idioms. |
| **Concurrency safety** | None (shared `Provider` slice). | Adds `sync.RWMutex`. | **Mutex** – required to avoid race conditions when the ticker goroutine updates health while the router reads it. |
| **Provider registration** | Hard‑coded slice inside `main`. | Global `RegisterProvider` helper. | **RegisterProvider** – lets other packages (e.g., tests, future plugins) add providers without touching the health file. |
| **Fail‑over routing** | Not addressed. | Explicit “skip unhealthy” note. | **Router guard** – we will modify the router to consult the health map before selecting a provider. |
| **Health endpoint** | Simple JSON map. | Structured JSON (`ProviderHealth`). | **Structured JSON** – gives callers more context (last‑check time) while still being easy to parse. |
| **Testing hooks** | None. | Clear test cases in description. | **Both** – we will provide a small test suite that drives the ticker, simulates failures, and checks the `/health` response. |

// File: /opt/axentx/thai-gateway/service/health.go
package service

import (
	"context"
	"encoding/json"
	"log"
	"net/http"
	"sync"
	"time"
)

// -------------------------------------------------------------------
// 1️⃣  Data structures
// -------------------------------------------------------------------

// ProviderHealth holds the runtime health status of an AI provider.
type ProviderHealth struct {
	Name      string    `json:"name"`       // friendly identifier
	URL       string    `json:"url"`        // base URL (no trailing slash)
	Healthy   bool      `json:"healthy"`    // true if last ping succeeded
	LastCheck time.Time `json:"last_check"` // timestamp of the most recent ping
}

// -------------------------------------------------------------------
// 2️⃣  Global registry (protected by RWMutex)
// -------------------------------------------------------------------
var (
	healthMu      sync.RWMutex               // protects the map below
	providerHealth = map[string]*ProviderHealth{} // key = provider name
)

// RegisterProvider is called once (usually from main) for every backend.
func RegisterProvider(name, baseURL string) {
	healthMu.Lock()
	defer healthMu.Unlock()
	providerHealth[name] = &ProviderHealth{
		Name:      name,
		URL:       baseURL,
		Healthy:   true,               // assume healthy until proven otherwise
		LastCheck: time.Now(),
	}
}

## 1. Acceptance Criteria
1. **Health Ping Frequency** – Gateway must send a health‑check request to each AI provider every **30 seconds** (±5 s jitter allowed).
2. **Unhealthy Provider Marking** – A provider that fails to respond within the timeout or returns a non‑2xx status is marked **unavailable** and excluded from routing decisions.
3. **Health Endpoint Exposure** – The `/health` endpoint must return a JSON payload containing the status (`healthy`/`unhealthy`) of every provider and the overall gateway health.
4. **Fail‑over Timing** – When a provider becomes unhealthy, the gateway must switch to an alternate healthy provider within **5 seconds** (including detection, marking, and routing update).
5. **Recovery** – A previously unhealthy provider that becomes healthy again must be re‑added to the routing pool automatically.
6. **Graceful Degradation** – If all providers become unhealthy, the gateway should return a 503 Service Unavailable on all client requests.
7. **Logging & Metrics** – Each health check attempt and result must be logged and exposed via Prometheus metrics (`provider_health_status`, `provider_failover_duration`).
