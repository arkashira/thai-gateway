            time.Sleep(retryDelay)
            continue

        return resp, nil

REJECT: The proposed changes contain multiple compile‑time errors that would prevent the code from building and running.

- `time.Duration(backoffMultiplier^attempt)` uses the bitwise XOR operator (`^`) instead of exponentiation; Go has no exponent operator for integers, so this expression is invalid.
- `httpClient` is referenced but never declared or imported; the code will not compile unless a client variable is defined or `http.DefaultClient` is used.
- `errors` is used but not imported, causing a missing package error.
- The retry loop uses `attempt <= maxRetries`, which will execute one more iteration than intended; the logic should be `< maxRetries` or `<= maxRetries-1`.
- The function signature and usage of `provider string` may not match the existing provider representation in the project, potentially causing type mismatches elsewhere.

These issues must be addressed before the change can be considered for integration.

## 1. Acceptance Criteria
1. **Retry Count** – The gateway must attempt to resend a failed request to the same provider **exactly 3 times** before giving up.  
2. **Exponential Backoff** – The delay between retries follows the sequence **100 ms, 200 ms, 400 ms** (or a configurable base × 2ⁿ).  
3. **Provider Fallback** – After the third failed retry, the request is automatically routed to the **next healthy provider** in the balancer’s pool.  
4. **Logging** – Each failed attempt (including the final failure) is logged with: timestamp, provider ID, request ID, HTTP status/error, and retry attempt number.  
5. **Metrics** – The gateway exposes Prometheus metrics `gateway_retry_total` (counter) and `gateway_fallback_total` (counter) with labels `provider` and `outcome`.  
6. **Idempotency Safety** – Retries are only performed for **idempotent HTTP methods** (GET, HEAD, OPTIONS). Non‑idempotent methods are forwarded directly to the next provider after the first failure.  
7. **Configuration** – All retry parameters (max attempts, base backoff, jitter) are configurable via `gateway.yaml` and validated at startup.
