## review — reviewer @ 2026-06-03T04:22:06.339978Z

REJECT: The proposed implementation contains several fundamental issues that will cause the service to fail at runtime.

- **Incorrect Prometheus client usage** – `promClient.register.counter` and `promClient.register.histogram` do not exist. Counters and histograms must be created with `new promClient.Counter({...})` or `new promClient.Histogram({...})`. As written, `req.metrics.requestCount` etc. will be `undefined`, causing `req.metrics.requestCount.inc()` to throw.
- **Metrics duplication and scope** – The middleware creates new metric instances for every request, which defeats the purpose of global counters/histograms and will lead to memory leaks and inaccurate data. Metrics should be defined once (as in `prometheus.js`) and imported where needed.
- **Mounting a full Express app under `/metrics`** – `prometheusMetrics` is an `express()` instance that already defines a `/metrics` route. Using `app.use('/metrics', prometheusMetrics)` results in the endpoint being reachable at `/metrics/metrics`, not the intended `/metrics`. A router or plain handler should be exported instead.
- **Reference to `req.provider`** – The logger middleware accesses `req.provider` when creating the `providerUsage` counter label, but this property is never set, leading to a label with value `undefined` or a runtime error.
- **Missing default metric collection** – While not a blocker, the code does not enable `collectDefaultMetrics()`, which is typically required for a useful Prometheus export.

These problems prevent the code from starting and from exposing correct metrics, so the change must be rejected until they are addressed.

## 1. Acceptance Criteria
1. **Metrics endpoint reachable** – `GET /metrics` returns HTTP 200 with `Content-Type: text/plain; version=0.0.4`.
2. **Request count metric** – `gateway_requests_total` increments by 1 for every incoming request (including health‑checks) and is labeled with `method`, `route`, and `status_code`.
3. **Error rate metric** – `gateway_errors_total` increments only for responses with status ≥ 500 and is labeled with `route`.
4. **Response time metric** – `gateway_response_duration_seconds` records a histogram of request latency, with default buckets `[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]`.
5. **Provider usage metric** – `gateway_provider_requests_total` increments per successful request and is labeled with `provider` (e.g., `stripe`, `paypal`).
6. **Request‑ID logging** – Every incoming HTTP request receives a unique `X-Request-ID` header (generated if missing) and the same ID is attached to all log entries for that request.
7. **Metrics are not exposed on non‑production environments** – When `NODE_ENV !== "production"` the `/metrics` route returns HTTP 404.

All criteria are measurable via automated tests.
