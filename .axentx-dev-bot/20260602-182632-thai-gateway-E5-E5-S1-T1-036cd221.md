### 1. Feature Overview  
**Thai‑Gateway Reproducibility Enhancer** is a targeted module that augments the existing Thai‑Gateway platform with automated reproducibility assessment for Thai‑language open‑source projects. It provides a reproducibility scorecard, actionable remediation steps, and integrates directly into the Thai‑Gateway CI pipeline.

---

### 2. Scope & Deliverables  

| Deliverable | Description |
|-------------|-------------|
| **Reproducibility Analyzer** | A Python library that parses a Thai‑language repo, extracts metadata, and evaluates reproducibility criteria. |
| **Scorecard API** | REST endpoint `/thai-gateway/repro-score` that accepts a repo URL and returns JSON scorecard. |
| **Dashboard Widget** | React component that displays the scorecard within Thai‑Gateway’s project view. |
| **CI Plugin** | GitHub Action `thai-gateway/repro-check` that runs on push/PR and posts comments with scorecard. |
| **Remediation Engine** | Rule‑based engine that maps low‑score items to concrete GitHub issues or PR templates. |
| **Data Store** | PostgreSQL table `repro_scores` to persist historical scores and trend analysis. |

---

### 3. Implementation Details  

#### 3.1 Analyzer Logic  
1. **Repository Cloning** – Use `gitpython` to clone the repo into a sandboxed Docker container.  
2. **Metadata Extraction**  
   - `pyproject.toml`, `setup.py`, `requirements.txt` → dependency graph.  
   - `README.md`, `CONTRIBUTING.md` → documentation completeness.  
   - `tests/` directory → test coverage via `coverage.py`.  
   - CI config files (`.github/workflows/*.yml`, `.gitlab-ci.yml`) → pipeline robustness.  
3. **Reproducibility Checks**  
   | Check | Implementation |
   |-------|----------------|
   | **Documentation** | NLP token count & keyword density; flag missing sections. |
   | **Build System** | Verify presence of `Makefile` or `build.gradle`; run `./build.sh` in container. |
   | **Test Coverage** | Run `pytest --cov` and parse coverage report. |
   | **Dependencies** | Resolve with `pipdeptree`; check for pinned versions. |
   | **Version Consistency** | Compare `__version__` in code vs `setup.py`. |
   | **CI Reliability** | Parse workflow logs; detect failures, flaky tests. |
   | **Community Engagement** | Count open issues, PRs, and recent activity via GitHub API. |
4. **Scoring Algorithm** – Weighted sum (weights tuned via A/B testing):  
   - Documentation 15%  
   - Build 10%  
   - Tests 25%  
   - Dependencies 10%  
   - Version 5%  
   - CI 20%  
   - Community 15%  
5. **Remediation Mapping** – For each low‑score dimension, generate a GitHub issue template with suggested actions (e.g., “Add `tests/` directory”, “Pin dependencies in `requirements.txt`”).

## 1. Acceptance Criteria
1. **Env‑var precedence** – When a provider endpoint or rate‑limit value is supplied via an environment variable, the gateway must use that value, overriding any value present in the YAML config file.  
2. **Supported variables** – The gateway must recognize the following environment variables:
   - `THAI_GATEWAY_PROVIDER_ENDPOINT` – string URL of the provider API.  
   - `THAI_GATEWAY_RATE_LIMIT` – integer number of requests per minute.  
   - `THAI_GATEWAY_TRANSLATION_API_KEY` – string API key for the translation service (advanced setting).  
3. **Default values** – If an environment variable is not set, the gateway falls back to the value defined in `config.yaml`.  
4. **Type validation** – Invalid values (e.g., non‑numeric rate limit) must cause a clear error at startup.  
5. **Logging** – On startup the gateway logs which source (env or file) supplied each configuration value.  
6. **Docker compatibility** – The Docker image must expose the environment variables and allow overriding them at container run time.  
7. **Documentation** – README contains a concise section showing how to set the three env vars and how they override the YAML file.
