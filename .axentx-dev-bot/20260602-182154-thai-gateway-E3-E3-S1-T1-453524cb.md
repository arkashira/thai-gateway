## review — reviewer @ 2026-06-03T05:50:09.613085Z

REJECT: The change contains several fundamental issues that will prevent the application from starting and exposing metrics.

- **Incorrect library usage** – `prometheus-client` and its `Client` API (`createMetric`, `getMetrics`) do not exist in the Node ecosystem; the standard library is `prom-client` with a different API (`Counter`, `Histogram`, `Gauge`, `register.metrics()`).
- **Wrong relative import** – `src/routes/metrics.js` does `require('./metrics')`, which resolves to `src/routes/metrics.js` itself, causing a circular require and a missing module error. It should import `../metrics`.
- **Route mounting creates duplicate path** – In `src/routes.js` the code does `router.use('/metrics', metricsRouter);` while `metricsRouter` already defines a `GET /metrics` handler. This results in the endpoint being reachable at `/metrics/metrics` instead of `/metrics`.
- **Metrics are never updated** – The defined counters/histograms/gauges are never incremented or observed anywhere in the code, so the exposed values will always be zero, defeating the purpose of monitoring.
- **Missing export of the router** – The `metricsRouter` file exports the router, but the main router expects a router that handles the root path; the current structure is inconsistent with typical Express patterns.  

These blockers must be addressed before the feature can be considered functional.

## 1. Acceptance Criteria  

1. **Endpoint availability** – `GET /metrics` returns **HTTP 200** within 200 ms.  
2. **Prometheus content type** – Response header `Content-Type` is exactly `text/plain; version=0.0.4`.  
3. **Required metric families** – The body contains the following metric families in Prometheus exposition format:  
   - `gateway_total_requests` (type: counter)  
   - `gateway_provider_latency_seconds` (type: histogram, label `provider`)  
   - `gateway_remaining_quota` (type: gauge, label `provider`)  
4. **Label correctness** – All registered providers appear as a `provider` label; label values are properly escaped (e.g., spaces → `\ `, backslashes → `\\`).  
5. **Metric accuracy after traffic** – After a simulated request flow (e.g., 5 requests to provider “alpha”, 3 to “beta”), the counters, histograms, and gauges reflect the exact numbers and latency buckets.  
6. **Scrape‑ability** – The metric payload parses without error using the official Prometheus text parser (`prom-client`’s `register.metrics()`), confirming syntactic validity.  
7. **Graceful handling of edge conditions** – The endpoint does **not** crash or return 5xx when:  
   - No providers have been registered.  
   - Quota for a provider is negative or exceeds `Number.MAX_SAFE_INTEGER`.  
   - Latency values are `NaN` or `Infinity`.
