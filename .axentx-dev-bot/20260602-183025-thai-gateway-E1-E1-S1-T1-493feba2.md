## 1. Acceptance Criteria
1. **Least‑congested selection** – For every incoming request the gateway queries the current usage percentage of all registered AI providers and routes the request to the provider with the *lowest* usage value.  
2. **Quota‑aware fallback** – If the selected provider’s usage exceeds its configured quota (e.g., 90 %), the gateway must retry with the next‑lowest‑usage provider **within 200 ms** of the original request start.  
3. **Deterministic tie‑break** – When two or more providers share the same lowest usage, the gateway selects the provider with the smallest provider‑ID (lexicographically) to ensure deterministic routing.  
4. **Logging** – Every routing decision is logged with:  
   - `request_id` (UUID)  
   - `selected_provider_id`  
   - `timestamp` (ISO‑8601)  
   - `usage_percentage` of the selected provider at decision time  
   - `fallback_attempts` (0 if first choice succeeded)  
5. **Metrics exposure** – The gateway exposes a Prometheus‑compatible metric `gateway_provider_selection_seconds` that records the latency of the selection algorithm (including any fallback retries).  
6. **Graceful degradation** – If *all* providers exceed quota, the gateway returns HTTP 503 with a JSON body `{ "error": "No available provider" }` and logs the failure.  
7. **Thread‑safety** – The provider selection algorithm must be safe under concurrent requests (no data races, consistent view of usage metrics).
