**Final, ready‑to‑use Dockerfile**

Create the file **`/opt/axentx/thai-gateway/Dockerfile`** with the exact contents below:

```dockerfile
# ---------- Build stage ----------
FROM node:18-alpine AS builder
WORKDIR /app

# 1️⃣ Copy only the package manifests – this lets Docker cache the dependency install
COPY package*.json ./

# 2️⃣ Install **only** production dependencies (no dev deps)
RUN npm ci --only=production

# 3️⃣ Copy the rest of the source code
COPY . .

# ---------- Runtime stage ----------
FROM node:18-alpine

# 4️⃣ Create a non‑root user for security
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nextjs -u 1001 -G nodejs

WORKDIR /app

# 5️⃣ Copy the compiled node_modules and source from the builder stage
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app . .

# 6️⃣ Switch to the non‑root user
USER nextjs

# 7️⃣ Expose the port the gateway listens on (default 3000)
EXPOSE 3000

# 8️⃣ Health‑check – poll the /health endpoint; fail if it does not return 200
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

# 9️⃣ Start the gateway
CMD ["node", "server.js"]
```

### **3. Integration Tests**

**Happy Path:**

1. **Build and run container, verify service health:**
   - Build image → Run container in background → Wait 5s → Send `GET /health`
   - Expect: HTTP 200, non-empty response body, container logs show "service started"

2. **Verify port exposure and mapping:**
   - Run `docker port <container>` → Expect `8080/tcp` mapped to host port (e.g., 8080)
   - `curl http://localhost:8080/health` returns 200

3. **Verify deployment script idempotence:**
   - Run `deploy.sh` twice → Second run gracefully handles existing container (stops/removes before relaunch)
   - Service remains reachable after second run

4. **Verify minimal image footprint:**
   - `docker inspect` the image → Confirm base is `alpine` or `distroless`
   - No shells (`/bin/sh`, `/bin/bash`) present unless required

5. **Verify container starts with non-root user:**
   - `docker exec <container> ps aux` → Process runs as UID > 1000
   - Security best practice enforced

**Edge Cases:**

1. **Build fails with dirty context (unintended files):**
   - Add `.env` and `__pycache__` to context → Confirm `.dockerignore` excludes them
   - Verify image size does not increase unexpectedly

2. **Port conflict during deployment:**
   - Manually bind host port 8080 → Run `deploy.sh`
   - Expect graceful error handling or auto-cleanup + exit code 1

3. **Health check fails (service crash):**
   - Modify `deploy.sh` to start with invalid args → Container runs but `/health` returns 500
   - Expect logs to indicate startup failure; health check must fail

### **4. Risk Register**

| Risk | Detection Method | Mitigation |
|------|------------------|----------|
| Container runs as root | `docker exec` + `ps aux` | Enforce `USER 1001` in Dockerfile |
| Sensitive files included in image | `docker save` + `tar -tvf`, or `dive` tool | Use `.dockerignore` with `**/.git`, `**/*.env`, `**/secrets` |
| Base image has critical CVE | Scan with `grype` or `trivy` | Use `alpine:latest` or `gcr.io/distroless/static` |
| Health check flakiness | Repeated `curl` with retries | Implement readiness delay or retry logic |
| Deployment script lacks cleanup | `docker ps -a` shows stale containers | Script must `docker stop && docker rm` prior instance |
| Port not exposed in Dockerfile | `docker inspect` shows no ExposedPorts | Add `EXPOSE 8080` |
| Build depends on external services (e.g., git clone in Dockerfile) | Build fails offline | Vendor deps or use multi-stage with cache |
