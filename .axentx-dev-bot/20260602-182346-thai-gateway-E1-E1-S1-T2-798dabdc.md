## 1. Acceptance Criteria

- **AC1** – `POST /config` on the Thai Gateway accepts a JSON array of provider objects (`{name, url, api_key}`) and registers all of them in the in-memory `ProviderRegistry` located at `/opt/axentx/thai-gateway/src/services/provider_registry.rs`.
- **AC2** – Each provider's health is checked within **5 seconds** (hard timeout) of being added/updated; the resulting `healthy: bool` is stored alongside the provider record and returned by `GET /providers`.
- **AC3** – Duplicate provider names within a single `POST /config` payload, or duplicates against existing registry entries, are rejected with HTTP `409 Conflict` and a descriptive JSON error body.
- **AC4** – Registry CRUD operations (`add`, `get`, `remove`, `list`) behave correctly and are the single source of truth for the gateway's routing layer.
- **AC5** – Every mutation (`add`/`remove`/bulk `POST /config`) is persisted to `/opt/axentx/thai-gateway/config/providers.json`; a gateway restart rehydrates the registry from this file before accepting traffic.
- **AC6** – File I/O failures (permission denied, disk full, corrupt JSON) and network failures (DNS, TCP refused, timeout) are caught, logged via the gateway's structured logger, and surfaced as non-`5xx` errors where applicable (`400`/`409`/`503`).
- **AC7** – The gateway continues serving requests via healthy providers even when one or more providers are marked unhealthy (no cascade failure).
