**Feature:** Thai‑Gateway Payment Integration  

**Description:**  
Add a native Thai payment gateway to the platform, enabling merchants to accept popular local payment methods (e.g., PromptPay, QR‑Pay, and major Thai bank card processors). The gateway will handle transaction routing, currency conversion, and compliance with Thai financial regulations, providing a seamless checkout experience for Thai consumers while giving merchants real‑time settlement reporting.

**Implementation Steps**

1. **Requirements & Compliance**
   - Gather regulatory requirements from the Bank of Thailand and local payment schemes.
   - Define supported payment methods: PromptPay QR, QR‑Pay (e.g., TrueMoney, Alipay Thailand), and major Thai card networks (Visa, MasterCard, JCB).
   - Establish PCI‑DSS and GDPR compliance checkpoints.

2. **Connector Architecture**
   - Design a modular connector layer (`thai_gateway_connector`) exposing a unified API (`/payments/initiate`, `/payments/status`, `/payments/refund`).
   - Implement adapters for each payment method:
     - **PromptPayAdapter** – QR code generation, OTP verification.
     - **BankCardAdapter** – Tokenization via hosted fields, 3‑DS flow.
     - **QRPayAdapter** – QR code generation for third‑party wallets.

3. **Transaction Flow**
   - **Initiate**: Merchant sends order details → gateway creates a transaction ID, selects payment method, returns a payment URL/QR.
   - **Authorize**: Customer completes payment via chosen method; gateway receives callback/webhook.
   - **Confirm**: Verify signature, update transaction status, notify merchant via webhook.
   - **Settle**: Daily batch settlement to merchant’s Thai bank account, with optional auto‑conversion to USD/EUR.

4. **Security & Fraud Controls**
   - Enforce TLS 1.3 for all external calls.
   - Validate webhook signatures using HMAC‑SHA256.
   - Integrate with a fraud‑risk engine (e.g., Sift) for velocity checks and IP/geolocation analysis.
   - Store only tokenized card data; raw PAN never touches our servers.

5. **Dashboard & Reporting**
   - Extend the existing merchant dashboard with a “Thai‑Gateway” tab:
     - Real‑time transaction list (status, amount, method).
     - Daily/weekly settlement reports with currency conversion details.
     - Refund and dispute management UI.
   - Exportable CSV/Excel reports for accounting.

6. **Testing & Validation**
   - Unit tests for each adapter (mocked API responses).
   - End‑to‑end integration tests using sandbox environments provided by PromptPay and Thai banks.
   - Load testing to verify handling of up to 5,000 concurrent checkout sessions.

7. **Deployment**
   - Containerize the connector (Docker) and deploy via Kubernetes with autoscaling.
   - Feature flag to enable/disable per‑merchant.
   - Rollout plan: beta to a limited set of Thai merchants → full release after 2 weeks of monitoring.

**Benefits**
- **Market Expansion:** Opens access to Thailand’s $70 B e‑commerce market.
- **Higher Conversion:** Local payment options reduce cart abandonment for Thai shoppers.
- **Compliance Assurance:** Built‑in regulatory checks keep merchants audit‑ready.
- **Revenue Growth:** Transaction fees from the new gateway create a recurring revenue stream.

**Safety & Compliance**
- All data in transit encrypted; no sensitive card data stored.
- PCI‑DSS validated service provider contracts for card processing.
- Regular security audits and automated vulnerability scans.

### 1. **Acceptance Criteria**

- [ ] On first API request, a unique API key is auto-generated and returned in the response header `X-API-Key`.  
- [ ] Each API key is bound to a unique platform identifier (e.g., `platform_id`), and cannot access data or endpoints of other platforms.  
- [ ] Rate limiting (e.g., 100 requests/minute) is applied per platform via its API key; limits are not shared across platforms.  
- [ ] A successful onboarding README exists at `/docs/thai-gateway/onboarding.md` with step-by-step instructions, curl examples, and security best practices.  
- [ ] API keys are transmitted over HTTPS only; attempts over HTTP result in 403 Forbidden.  
- [ ] Invalid or missing API keys return 401 Unauthorized with clear error messaging.

### 3. **Integration Tests**

**Happy Paths:**
1. New platform sends first request → receives API key, is rate-limited independently, and accesses gateway successfully.
2. Platform uses API key in subsequent requests → full access to its scoped endpoints (e.g., `/orders`, `/webhooks`).
3. Onboarding README is rendered in GitHub and contains working curl examples with placeholder substitution.

**Edge Cases:**
1. Two platforms behind same public IP (NAT) → still receive distinct keys and platform IDs based on fingerprinting (User-Agent + headers).
2. Malformed Authorization header (e.g., `Bearer xyz` where xyz not a valid key) → returns 401 with `invalid_api_key`.
3. Replayed API key from decommissioned platform → rejected if key is revoked or platform marked inactive.
